# Issue Management

This repo uses an agent-facing issue workflow. Agents manage issue state through comments and lifecycle labels only.

## Lifecycle labels

| Role | Label | Meaning |
| --- | --- | --- |
| review | `agent:review` | Needs an agent readiness review. |
| ready | `agent:ready` | Ready for an implementation agent to claim. |
| claimed | `agent:claimed` | Claimed by an agent. |
| blocked | `agent:blocked` | Not actionable by agents until a human changes the label. |

Exactly one lifecycle label should be present on an issue. If multiple lifecycle labels are present, agents must stop and ask which state is intended.

## Authority boundaries

Agents may:

- inspect issues and repository context;
- post comments with the AI disclosure;
- add and remove lifecycle labels automatically;
- draft new issues;
- publish new issues when explicitly asked;
- update `.out-of-scope/` after an explicit human out-of-scope/rejection decision.

Agents must not:

- edit issue bodies;
- manage assignees;
- create missing tracker labels unless a human explicitly permits it;
- close issues;
- mutate non-agent labels unless explicitly asked.

## Comment disclosure

Every agent-authored issue comment starts with:

> *This was generated by an AI agent during issue management.*

## Readiness review

An issue is ready for agents when:

- the goal is clear;
- the done state is verifiable through acceptance criteria or an obvious validation path;
- the scope is not obviously too large for one agent session;
- known dependencies and blockers are stated, or none are found;
- relevant code or documentation areas can be identified;
- no unresolved product, design, or security decision is required;
- validation is known.

Review outcomes:

- Ready: move `agent:review` to `agent:ready` and comment with the reasoning.
- Blocked: move `agent:review` to `agent:blocked` and comment with the blocker and needed human action.

Agents do not re-review `agent:blocked` issues. A human moves `agent:blocked` back to `agent:review` when another review is wanted.

## Claiming and stale claims

Agents may claim issues labeled `agent:ready`.

Claim comments include:

```md
## Agent claim

Claimed at: <ISO timestamp>

Plan:
- <short plan>
```

The stale claim window is 24 hours. If the latest parseable claim is older than that and there is no later claim, progress, PR, or completion activity, agents may move `agent:claimed` back to `agent:ready` after posting a stale-claim comment.

Progress comments are optional. If work continues past the stale window, the working agent should post a progress or renewed claim comment.

## Completion

When work is completed, agents post a completion handoff comment with links to PRs, branches, or commits if available, summarize validation and remaining risks, then remove all lifecycle labels. Humans close issues.

## Issue authoring

Agents draft issue breakdowns first unless the user explicitly says to publish. Published issues default to `agent:review`; use `agent:ready` only when explicitly requested or after readiness review.

Agent-ready issues should include:

- goal;
- context;
- acceptance criteria or verifiable done state;
- suggested starting points;
- validation;
- known blockers or decisions.

Specs should be behavior-first. File paths are allowed as suggested starting points, but line numbers and procedural edit instructions should not be the acceptance contract.

## Out-of-scope decisions

Agents may read `.out-of-scope/*.md` and surface prior matching decisions during review.

Agents may create or update `.out-of-scope/` only after an explicit human decision that a request is out of scope, rejected, or wontfix. After recording the decision, agents comment on the issue and remove lifecycle labels. Humans close issues.
